1.2 Project Goal

The objective of this project is to design and implement a comprehensive automated end-to-end test suite for GOG.com, a major DRM-free PC gaming platform operated by CD Projekt. The suite validates the core user-facing functionality of the site — browsing, searching, authentication, account creation, game pages, genre discovery, and cart/checkout — through 51 automated test cases organized across 8 independent test classes.

By automating these tests, the team aims to:

  • Detect regressions in critical user flows (login, search, checkout) before they reach end users.
  • Verify that GOG's Angular Single Page Application renders required UI elements correctly across sessions.
  • Demonstrate the application of Selenium WebDriver with TestNG in a real-world, production environment.
  • Practice handling automation challenges such as Angular/React SPAs, GDPR consent modals, dynamic elements, and bot-detection headers.

1.3 System & Test Summary

System Under Test

GOG.com (Good Old Games, https://www.gog.com) is a global digital game distribution platform with a catalog of over 6,000 DRM-free titles. The site is built on a React/Angular hybrid architecture with a dedicated authentication subdomain (login.gog.com) powered by Angular.

Test Classification

All tests are System / End-to-End tests: each test class exercises a specific page or feature area of the GOG.com production site using a real Chrome browser controlled by Selenium WebDriver. The full suite is integrated via testng.xml, making it a combined integration run of all 8 modules.

Test Design Approach

Each test class extends a shared BaseTest class that initializes and tears down ChromeDriver once per class. A @BeforeMethod in each class navigates to the relevant page before every test. Tests use CSS selectors, XPath, and WebDriverWait with a 20-second timeout. Shared helpers (cookie consent dismissal, page-load wait, element presence check) are centralized in TestUtils.

2. Work Conducted

2.1 Overview of Work

Test Planning

The team selected GOG.com as the system under test because it is a publicly accessible, feature-rich e-commerce platform with documented automation challenges: an Angular SPA authentication layer, a React-based storefront, aggressive bot-detection headers, and a GDPR consent modal on first visit. The team identified 8 functional areas to test and mapped each to a test class.

Test Development

Each test class was authored in Java using Selenium WebDriver 4.18.1 and TestNG 7.9.0. Key challenges addressed during development:

  • Angular SPA (login.gog.com): The auth page is served as an Angular application. Direct DOM clicks on form triggers do not fire Angular zone.js change detection, so tests were designed to verify element presence and accessibility rather than form submission.
  • React-rendered text: WebElement.getText() returns empty strings for React-rendered elements in some page configurations. Solution: use getAttribute("textContent") and visibilityOfElementLocated.
  • Cart subdomain DNS: cart.gog.com does not resolve reliably in automated environments. Solution: navigate dynamically from www.gog.com via a header link in @BeforeMethod.
  • Cookie consent interception: GOG's GDPR banner overlaps clickable elements. TestUtils.dismissCookieConsent() dismisses it first using a JavaScript click fallback when the standard Selenium click is intercepted.
  • UTF-8 BOM: All source files were stripped of the UTF-8 BOM (0xEF 0xBB 0xBF) that the original editor introduced, which caused javac compilation errors.

Integration of Test Classes

All 8 test classes are integrated in testng.xml, which defines the suite name "GOG Selenium Test Suite" and runs each class as a separate named test in sequence with parallel="none". The full suite runs in approximately 15–20 minutes.

Execution and Reporting

Tests are executed via mvn test. Maven Surefire Plugin 3.2.5 generates per-class XML reports in target/surefire-reports/. Final result: 51 tests run, 0 failures, 0 errors, 0 skipped.

4. Motivation

GOG.com serves millions of users globally. Regressions in core flows — login, search, or checkout — can directly impact revenue and user trust. Manual testing at scale is slow and error-prone; an automated suite provides repeatable, objective verification of critical functionality.

Issues our tests help prevent or uncover:

  • Broken authentication: LoginPageTest verifies the sign-in trigger, forgot password link, and create account link are accessible. If a frontend deployment broke any of these, the test would immediately flag it.
  • Search regressions: SearchTest confirms that search returns game tiles with titles and cover images. A broken search would be caught before users encounter empty results.
  • Cart/checkout unavailability: CartPageTest verifies the cart page loads with proper navigation and HTTPS. Any misconfiguration of the cart subdomain would be detected.
  • Game page content missing: GamePageTest checks that the buy button, description, screenshots, and system requirements are all rendered. Missing sections would be caught automatically.
  • Navigation broken: NavigationTest verifies the GOG logo link, store navigation, and header persistence across pages.

How the suite improves system quality: The tests serve as a regression safety net. After any GOG frontend update, running mvn test produces a pass/fail report for all 51 test cases within 20 minutes — far faster than manual verification. The suite also documents the expected behavior of each page, serving as executable specifications.

Appendix B – Automation Constraints & Solutions

Constraint Solution Applied
Angular SPA (login.gog.com) – forms not revealed by DOM click Tests verify the trigger element presence rather than form interaction; JavaScript click used to avoid Angular routing issues
React-rendered text – getText() returns empty string Used getAttribute("textContent") and visibilityOfElementLocated()
cart.gog.com DNS not resolving in automated environment Navigate to cart dynamically from www.gog.com via header link in @BeforeMethod
GDPR cookie consent modal intercepts clicks TestUtils.dismissCookieConsent() dismisses with JS click fallback on ElementClickInterceptedException
GOG bot-detection headers ChromeOptions: --disable-blink-features=AutomationControlled; excludeSwitches: enable-automation; useAutomationExtension: false
signup URL (login.gog.com/signup) returns HTTP 404 SignupPageTest navigates to login.gog.com/login where the Create Account link is accessible
UTF-8 BOM in source files causing javac errors Stripped BOM bytes (0xEF 0xBB 0xBF) from all 11 Java files via PowerShell byte manipulation
